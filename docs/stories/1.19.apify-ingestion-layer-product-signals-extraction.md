## Story

**As a** platform engineer,  
**I want** a standalone Apify-backed ingestion layer that turns raw source data into `ProductSignals`,  
**so that** I can feed the existing scoring engine without touching the live MVP ingestion pipeline, API, dashboard, or persistence flow.

## Acceptance Criteria

1. A standalone top-level `ingestion/` package is added with `__init__.py`, `config.py`, `clients/apify_client.py`, `scrapers/base.py`, `scrapers/tiktok_shop.py`, `scrapers/tiktok_videos.py`, `transformers/product_transformer.py`, `transformers/metrics_transformer.py`, `pipeline.py`, `storage.py`, and `__main__.py`, and it stays isolated from `services/ingestion/`, `bin/radar.py`, `services/shared/config.py`, DB/API/dashboard code, and the live runtime control plane.
2. `ingestion/config.py` loads `APIFY_TOKEN` and ingestion settings from environment variables via `python-dotenv`; missing `APIFY_TOKEN` fails fast with a clear startup error, and keywords/cache settings remain configurable without hardcoding.
3. `ApifyClient` uses `httpx.AsyncClient` and exposes `run_actor()`, `wait_for_run()`, `get_dataset_items()`, and `run_and_collect()` with retry logic, exponential backoff on 5xx errors, timeout handling, and no token leakage in logs or exceptions.
4. `scrapers/base.py` defines the abstract scraper boundary; `TikTokShopScraper` and `TikTokVideosScraper` only fetch raw `list[dict]` payloads with the correct Apify input shapes and do not perform transformation work.
5. `compute_video_metrics()` and `transform_product()` remain pure and deterministic. In this story, `view_growth_7d` and `view_growth_3d` are treated as the share of recent plays over total plays for the sampled video set, clamped to `[0.0, 1.0]`, and `days_since_detected` is derived from the latest global cached signals by `product_id`, incrementing `+1` when a product reappears and starting at `0` when no history exists.
6. `IngestionPipeline.run()` processes keywords concurrently with `asyncio.gather()`, saves raw source payloads and final signals to local JSON cache, continues other keywords when one keyword fails, and returns a deterministic `list[ProductSignals]` ready for `score_batch()`.
7. `python -m ingestion` loads configuration from env, accepts keyword overrides and an output path, writes the final signals JSON, and prints the requested ingestion summary to stdout.
8. Tests cover the client retry and timeout behavior, scraper input/output contracts, transformer math and edge cases, cache roundtrip, pipeline concurrency and partial-failure handling, and the repo tooling includes `ingestion/` in lint/typecheck/build/package coverage if the existing command lists need a minimal update.

## Dev Notes

### Architecture Decision

- `ingestion/` is a new parallel package for external acquisition and signal synthesis. It does not replace or refactor the existing MVP bootstrap ingestion pipeline in `services/ingestion/`.
- The current live ingestion flow in `services/ingestion/service.py` remains the canonical path for persistence-backed bootstrap data.
- This story only produces `ProductSignals` for downstream scoring and keeps its own local JSON cache. It must not add DB writes, FastAPI routes, dashboard wiring, or scheduling.

### Existing Context

- `scoring/models.py` already defines `ProductSignals`, which is the required output of this layer.
- `services/ingestion/` already handles the MVP bootstrap ingestion and should remain unchanged in this story.
- `bin/radar.py` already owns the CLI control plane; this story must add a separate `python -m ingestion` entrypoint instead of threading Apify into the existing control plane.
- `docs/project-tracker.md` currently has all backlog work closed, so this story needs a new tracker entry and backlog task to keep the repo story-driven.

### Clarified Contracts

- `view_growth_7d` and `view_growth_3d` are defined here as recent-play share ratios for the sampled video set, clamped to `[0.0, 1.0]`, so the metrics stay deterministic and bounded.
- `load_latest_signals()` should use the most recent global signals cache file, not a per-keyword or per-source variant.
- `days_since_detected` should be based on `product_id` continuity in the latest cached signals: `0` on first sight, `+1` on reappearance.
- If one keyword fails during a pipeline run, the other keywords still complete and return usable signals.

### File Locations

- Create the new package under a top-level `ingestion/` directory.
- Add tests under `tests/`.
- Update `package.json` and `pyproject.toml` only if the existing validation/build commands must enumerate `ingestion/` explicitly.

### Constraints

- Keep this story limited to Apify ingestion, transformers, cache, pipeline, CLI, and tests.
- Do not add database models, API endpoints, dashboard wiring, scoring changes, ranking changes, or scheduling.
- Do not modify `services/ingestion/`, `bin/radar.py`, or `services/shared/config.py`.
- Do not implement the Bright Data adapter; leave the future boundary as a TODO in `base.py`.

### Testing

- Run `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build` before closing the story.
- Use async tests for `ApifyClient` and the pipeline.
- Cover empty inputs, 5xx retries, timeout/failure handling, cache roundtrip, and partial keyword failure behavior.
