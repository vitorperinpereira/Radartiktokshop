## Story

**As a** project operator,
**I want** the local runbook, testing guide, and source governance docs to reflect the now-live weekly flow and validation coverage,
**so that** I can bootstrap, run, validate, and troubleshoot the MVP locally without guessing which commands, fixtures, and source rules the repository actually supports.

## Acceptance Criteria

1. `docs/runbook-local.md` documents the current local-first operator flow with concrete steps for bootstrap, seeded weekly execution, persisted read inspection, report export, reset, and troubleshooting, aligned to the CLI/API/dashboard surfaces that actually exist in the repo after `CPR-022`.
2. `docs/testing.md` documents the project's active unit, integration, and smoke strategy with current commands, seed usage, and validation scope, reflecting the real coverage path for weekly run, API/dashboard reads, and report export rather than outdated placeholders.
3. `docs/source-governance.md` documents the current source modes, adapter and lineage expectations, fallback policy, and seed/mock usage in a way that matches the implemented ingestion flow and local validation practices.
4. Documentation remains CLI-first and local-first, does not invent live-connector requirements or publication workflow, and stays limited to the existing MVP behavior and repo artifacts.
5. Verification notes confirm the documented commands and flows align with the repository state after the coverage hardening in `CPR-022`.
