## Story

**As a** platform engineer,
**I want** a pure Python scoring engine module that implements the requested deterministic product scoring math,
**so that** the team can evaluate the new Creator Opportunity Score model in isolation without breaking the current weekly runtime pipeline.

## Acceptance Criteria

1. A deterministic scoring module is added as a standalone top-level `scoring/` package with `__init__.py`, `models.py`, `factors.py`, `aggregator.py`, and `calibration.py`, implementing the provided formulas, labels, and typed contracts without database or API dependencies.
2. The new engine exposes `ProductSignals`, `ProductScore`, `ScoringParams`, `default_params()`, `from_dict()`, `score_product()`, and `score_batch()` with full type hints, Pydantic v2 models, and docstrings on public functions.
3. Factor functions remain pure and side-effect free, using runtime parameters from `ScoringParams` rather than hidden constants for tunable math behavior.
4. The current runtime aggregation path in `services/scoring/aggregation.py` and its existing imports remain compatible; this story must add the new engine without replacing or renaming the live weekly-run pipeline.
5. Tests cover the requested factor behavior, batch `r_max` handling, acceleration bonus behavior, sorting, and one full product scoring path, and the repo quality gates pass.

## Dev Notes

### Architecture Decision

- The requested engine should be implemented as a parallel deterministic top-level `scoring/` package, not as a replacement for `services/scoring/aggregation.py`.
- Reason: the repository already has a live scoring pipeline, persistence contract, and tests tied to `services/scoring/aggregation.py` plus `docs/scoring-model.md`. Replacing that code in this story would create cross-story regressions outside the requested scope.
- The new engine is therefore an isolated scoring module that can be imported and tested independently now, while the current weekly runtime stays unchanged.

### Existing Context

- `services/scoring/aggregation.py` is the current live runtime aggregator used by the weekly pipeline and must remain stable.
- `services/scoring/__init__.py` currently exports the live aggregation APIs and should continue to do so after this story.
- `tests/unit/test_score_aggregation.py` covers the current runtime aggregator; the new tests should live alongside it without replacing those assertions.
- `docs/scoring-model.md` describes the current MVP scoring formula, which differs from this new requested engine. This story should not silently rewrite the live runtime model.

### File Locations

- Create the new engine under a top-level `scoring/` package.
- Add tests under `tests/`.

### Constraints

- Keep this story limited to the pure scoring engine and tests.
- Do not add database models, API endpoints, CLI commands, orchestration hooks, or dashboard wiring.
- Do not remove or break the existing `services/scoring/aggregation.py` runtime path.
- Prefer deterministic math from the Python standard library only; no NumPy dependency.

### Testing

- Run the requested factor and aggregator coverage for the new module.
- Re-run the existing repo quality gates before closing the story.
