## Story

**As a** project operator,
**I want** read-only FastAPI endpoints for weekly ranking, product detail, and pipeline/report history,
**so that** downstream operator surfaces can consume persisted weekly scoring output without querying the database directly.

## Acceptance Criteria

1. FastAPI exposes a read-only weekly ranking endpoint backed by persisted `product_scores` joined with `products`, returning the ranked weekly output ordered by `final_score` and including the product metadata and score/explainability fields needed by the current Weekly Radar surface.
2. FastAPI exposes a read-only product detail endpoint backed by `products` and `product_scores`, returning product metadata plus the persisted weekly score breakdown for one product, including agent sub-scores, `saturation_penalty`, `revenue_estimate`, `final_score`, `classification`, and the stored `explainability_payload` without recalculating scores.
3. FastAPI exposes read-only history endpoints backed by `pipeline_runs` and `reports`, returning persisted run/report identifiers, `week_start`, statuses, timestamps, `error_summary` when present, and report payload data or payload-derived metadata already stored in the database.
4. `apps/api/main.py` is extended beyond `/` and `/health` to mount the new read API surface while preserving the existing ops endpoints and keeping the API read-only for this story.
5. Tests cover ranking, product-detail, and history endpoint behavior using existing persisted tables/payloads, including ordering, empty-result behavior, and not-found cases where applicable.

## Dev Notes

### Implementation Context

- `docs/backlog.md` defines `CPR-018` under Epic 5 as "Build ranking, product detail, and report history APIs" with output limited to read endpoints for weekly output.
- `docs/project-tracker.md` marks the current phase as "API and report surfaces prep" and lists opening the next story for `CPR-018` as the next pending step.
- `docs/implementation-plan.md` places ranking endpoints, product detail endpoints, and report history endpoints in Phase 4 before Streamlit ranking/detail views and before the report payload builder.
- Story 1.10 completed deterministic final aggregation and persistence into `product_scores`, so this story should read stored scoring output rather than recalculate any scoring logic.
- `apps/api/main.py` currently defines only `/` and `/health`, making this the first story that expands the FastAPI surface beyond ops checks.

### Data and Persistence Notes

- `services/shared/db/models.py` already defines the persistence sources required for this story: `Product`, `ProductScore`, `PipelineRun`, and `Report`.
- `ProductScore` already contains `trend_score`, `viral_potential_score`, `creator_accessibility_score`, `saturation_penalty`, `revenue_estimate`, `final_score`, `classification`, and `explainability_payload`, which should be reused directly for ranking/detail responses.
- `ProductScore` is indexed by `(week_start, final_score)`, which aligns with the weekly ranking read path for ordered results.
- `Report` already persists `status`, `report_payload`, `published_at`, `created_at`, and `run_id`; this story should read those stored values as-is instead of generating new report content.
- `PipelineRun` already persists `week_start`, status, start/finish timestamps, input job IDs, config version, and `error_summary`; history responses should stay within fields already stored or directly derivable from existing tables.

### Consumer Surface Notes

- `apps/dashboard/pages/1_weekly_radar.py` is still a mock, but it clearly establishes the first consumer shape: a weekly ranking table with score-led ordering and simple filterable product metadata.
- `apps/dashboard/pages/2_product_drilldown.py` is also a mock and expects a product score breakdown plus explainability-style reasoning. This story should provide the persisted score and explanation contract only.
- `apps/dashboard/pages/3_pipeline_history.py` is a mock for recent run/history viewing. It should be fed from persisted run/report history metadata only; execution log storage is not defined in the current schema and should not be introduced here.
- The current product drilldown mock shows a content-angle hook, but this story should not add content-angle/report-builder behavior because that belongs to `CPR-020`.

### File Locations

- Keep `apps/api/main.py` as the FastAPI assembly entrypoint and extend it to include the new read API surface.
- Reuse `services/shared/db/session.py` for API-side database session wiring instead of introducing a second database bootstrap path.
- There is no existing API query/router module yet beyond `apps/api/main.py`, so introduce dedicated read modules in `apps/api/` and/or a focused read-service module under `services/` only as needed to keep handlers thin.
- Add API-focused regression coverage under `tests/` in the same style as existing integration coverage used for persistence and weekly-run stories.

### Constraints

- Keep this story limited to `CPR-018`.
- Do not add Streamlit dashboard implementation in this story; `CPR-019` owns the ranking and drilldown UI work.
- Do not add report builder, export payload generation, or publication workflow in this story; `CPR-020` owns that scope.
- Do not add new schema columns, migrations, or new persistence tables in this story.
- Do not invent execution-log storage or history fields that are not already stored or directly derivable from existing persisted records.
- Keep the FastAPI surface read-only for this story.

### Testing

- Add FastAPI endpoint tests for weekly ranking, product detail, pipeline run history, and report history using sqlite-backed persisted fixtures.
- Verify the ranking endpoint reads ordered `product_scores` data and returns only stored score/explainability values.
- Verify product detail returns the persisted score breakdown and stored explainability payload for a single product.
- Verify history endpoints cover both populated and empty datasets, plus not-found behavior where applicable.
- Verify `/` and `/health` remain available after the new read endpoints are mounted.
- Update `docs/project-tracker.md` after implementation.
