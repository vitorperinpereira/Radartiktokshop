## Story

**As a** project operator,
**I want** broader integration and smoke coverage for the CLI-first weekly run and its read surfaces,
**so that** I can validate the end-to-end seeded flow from ingestion through report export with confidence before relying on it for recurring local runs.

## Acceptance Criteria

1. The current smoke path validates the seeded CLI flow on a clean local database, covering `db-upgrade`, `ingest-mock`, `weekly-run`, and `export-report`, and asserting structured JSON output plus persisted weekly-report artifacts for a completed run.
2. Integration coverage validates that a completed seeded weekly run can be consumed through the existing persisted read surfaces, including pipeline/report history plus ranking and product-detail reads, without introducing new write endpoints or publication workflow behavior.
3. Dashboard-side regression/smoke coverage validates the current read-only operator surface against persisted weekly output, ensuring the repo's active dashboard boot/read path can access the latest ranking/detail data without crashing and without expanding into new dashboard features.
4. Coverage remains CLI-first and local-first: it must rely on existing seed profiles, current read contracts, and the existing persisted storage/runtime behavior rather than live connectors, remote services, or invented test fixtures outside the repo's current patterns.
5. Tests and verification notes cover both successful seeded execution and at least one failure/empty-path scenario where appropriate for the weekly run or downstream reads, keeping the scope limited to validation hardening for `CPR-022`.

## QA Results

### Gate Decision: CONCERNS

### Findings

1. Medium: The dashboard smoke check proves Streamlit became reachable, but not that the dashboard successfully read persisted weekly data. In `tests/smoke/test_cli_weekly_run.py`, the assertion at line 122 only checks that the returned HTML contains `"streamlit"`. Because `apps/dashboard/app.py` stops on `load_rankings()` errors after rendering `st.error(...)` at lines 14-17, this smoke path can still pass when the dashboard cannot load the ranking data required by AC 3.

### Traceability

- AC 1: Covered by the seeded CLI smoke flow through `db-upgrade`, `ingest-mock`, `weekly-run`, and `export-report`, plus persisted `reports` and `content_angles` assertions.
- AC 2: Covered by the existing integration reads for rankings, product detail, pipeline history, and report history.
- AC 3: Partially covered. Dashboard boot is validated, but the smoke assertion does not distinguish a successful ranking read from an error page.
- AC 4: Covered. Validation remains local-first and uses the current seed profile/runtime.
- AC 5: Covered for empty/failure read paths in integration/unit coverage; see concern above for dashboard success-path evidence quality.

### Evidence

- Targeted verification passed: `python -m uv run --cache-dir .uv-cache --frozen --no-sync pytest -q tests/smoke/test_cli_weekly_run.py tests/integration/test_api_read_endpoints.py tests/unit/test_dashboard_read_adapter.py` (`10 passed`).

### Recommendation

- Strengthen the dashboard smoke assertion so it proves the success branch executed, for example by checking for a ranking-derived marker such as `Latest scored week`, `Products scored`, or a seeded product title after the weekly run completes.
