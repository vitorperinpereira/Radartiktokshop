## Story

**As a** project operator,  
**I want** the real Streamlit dashboard read surface to be hardened against persisted backend data and its main interactions proven by tests,  
**so that** I can trust the landing, Weekly Radar, Product Drilldown, and Pipeline History flows when reviewing a seeded weekly run locally.

## Acceptance Criteria

1. `apps/dashboard/app.py` remains the official dashboard entrypoint and proves the persisted ranking success-path on load, not just Streamlit reachability, while staying read-only and CLI-first.
2. `apps/dashboard/pages/1_weekly_radar.py` continues to use the existing dashboard-side adapter/read-service path, and its live-data interactions are hardened and testable: category filtering, hide-high-saturation filtering, persisted product selection, and the `Open Product Drilldown` handoff through `st.session_state`.
3. `apps/dashboard/pages/2_product_drilldown.py` continues to use the existing adapter/read-service path and reliably resolves the selected product/week context from persisted data, rendering the correct detail, explainability, snapshot metadata, and empty/error/not-found states without dashboard crashes.
4. `apps/dashboard/pages/3_pipeline_history.py` stops using local mock data and reads the existing persisted pipeline-history backend surface already available in the repo, without adding dashboard write actions, new persistence flows, or unrelated product features.
5. Smoke/integration coverage proves the seeded CLI flow can be consumed through the real Streamlit dashboard success-path, including stronger evidence than simple HTML reachability, plus focused validation of the main dashboard interactions and pipeline-history read behavior.
6. The story remains scoped to the current MVP read surfaces only: `apps/frontend/`, the Jinja router/templates surface under `apps/dashboard/router.py` and `apps/dashboard/templates/`, offline `scoring/` and `ranking/`, and write/persistence workflow changes stay out of scope unless a minimal read-contract adjustment is strictly required.
7. If needed for operator correctness, `package.json` may receive a minimal dashboard entrypoint fix, but no broader tooling or runtime-control-plane redesign is introduced.

## Dev Notes

### Architecture Decision

- The official frontend surface for this story is the Streamlit dashboard under `apps/dashboard/`, not `apps/frontend/`.
- The safe path is to harden the existing read-only architecture:
  - `Streamlit page -> apps.dashboard.read_adapter -> services.reporting.read_service -> DB/session`
- Do not migrate dashboard reads to HTTP calls against FastAPI in this story. The FastAPI app exposes the same read contract in parallel, but the dashboard's current internal adapter path is the intended MVP surface.

### Existing Context

- `apps/dashboard/app.py` currently loads rankings through `load_rankings()` and can stop on adapter error before proving a success-path render.
- `apps/dashboard/pages/1_weekly_radar.py` already uses persisted live ranking data, frontend-side filters, `st.session_state`, and `st.switch_page()` for Product Drilldown handoff.
- `apps/dashboard/pages/2_product_drilldown.py` already uses persisted live ranking/detail data but lacks direct interaction-proof coverage for the selected product/week success-path.
- `apps/dashboard/pages/3_pipeline_history.py` is still mock-based even though the backend already exposes real pipeline history reads through the existing read surface.
- `tests/smoke/test_cli_weekly_run.py` currently proves dashboard reachability after a seeded CLI flow, but its HTML assertion is too weak to prove the success branch executed with persisted ranking data.
- `tests/unit/test_dashboard_read_adapter.py` already covers the adapter well for ranking/detail happy-paths and failure-paths; prefer extending current patterns before inventing a new framework.
- `tests/integration/test_api_read_endpoints.py` already validates the persisted read surface for ranking, product detail, pipeline history, and report history.

### File Locations

- Keep the real dashboard entrypoint in `apps/dashboard/app.py`.
- Implement landing and page changes under `apps/dashboard/`.
- Reuse `apps/dashboard/read_adapter.py`, `apps/api/main.py`, `apps/api/schemas.py`, and `services/reporting/read_service.py` as the current read-contract surfaces.
- Extend tests under `tests/unit/`, `tests/integration/`, and/or `tests/smoke/` using the repo's current `pytest` structure.
- Update `package.json` only if a minimal dashboard command fix is required.

### Constraints

- Keep this story limited to the current MVP dashboard/read-surface hardening scope.
- Do not add dashboard write actions, new publication/report workflows, new persistence models, or orchestration/scoring changes.
- Do not touch `apps/frontend/`.
- Do not rework the legacy Jinja dashboard surface in `apps/dashboard/router.py` or `apps/dashboard/templates/*`.
- Do not change offline `scoring/`, offline `ranking/`, or report-builder persistence flows unless a minimal read-contract blocker is discovered.
- Preserve `CLI First -> Observability Second -> UI Third`: seeded CLI execution remains the primary acceptance path, with the dashboard validated as a downstream read surface.

### Testing

- Strengthen the seeded smoke test so it proves the real dashboard success-path executed with persisted ranking data after the CLI flow completes.
- Add or extend focused regression coverage for Weekly Radar filters, drilldown handoff/session state, Product Drilldown selection continuity, and Pipeline History live reads.
- Reuse the existing `smoke` seed profile and current local test patterns; do not introduce remote dependencies or a parallel test framework unless absolutely necessary.
- Run `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build` before closing the story.
