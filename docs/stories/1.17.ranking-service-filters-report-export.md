## Story

**As a** platform engineer,
**I want** a standalone ranking service that consumes the offline scoring engine and exports weekly rankings in JSON and CSV,
**so that** I can generate ranked product reports without touching the live API, database, dashboard, or runtime control plane.

## Acceptance Criteria

1. A standalone top-level `ranking/` package is added with `__init__.py`, `service.py`, `filters.py`, `report.py`, `models.py`, and `cli.py`, and it consumes the existing top-level `scoring/` package instead of reimplementing scoring math.
2. The new ranking layer exposes `RankingService`, `RankingFilters`, `apply_filters()`, `RankingEntry`, and `RankingReport` with typed, Pydantic-based contracts and stateless service behavior.
3. `RankingService.generate_ranking()` scores an in-memory list of `ProductSignals`, applies optional filters, sorts and ranks the results, computes `estimated_weekly_commission`, and produces a fully populated `RankingReport`.
4. `ranking/report.py` can export `RankingReport` to pretty-printed JSON and CSV using stdlib only, and can deserialize a saved JSON report back into the Pydantic report model.
5. The CLI entrypoint `python -m ranking.cli` loads product signals from JSON, applies CLI filters, writes JSON and CSV outputs into an output directory, and prints the requested summary without wiring into `bin/radar.py`, API, DB, dashboard, or report persistence.
6. Tests cover ranking generation, filtering, week-label formatting, commission math, JSON/CSV export, roundtrip loading, sorting, and empty-input behavior, while the repo quality gates remain green.

## Dev Notes

### Architecture Decision

- `ranking/` should remain a standalone top-level package, isolated from the live runtime surfaces.
- It must consume the already-delivered `scoring/` package for deterministic scoring and must not duplicate scoring math.
- This story is intentionally offline and in-memory: it should not touch `bin/radar.py`, `apps/api`, `services/reporting`, `apps/dashboard`, persistence, or scheduling.

### Existing Context

- `scoring/` already provides the scoring engine and must be reused as-is.
- `services/reporting/read_service.py` and `services/reporting/builder.py` already own the live persisted ranking/report surfaces and should remain unchanged in this story.
- `apps/api/main.py` and `apps/dashboard/read_adapter.py` already expose the live read surfaces for the current MVP; the new offline ranking layer should not integrate with them.
- The new CLI should live under `ranking/cli.py` and be invoked directly as `python -m ranking.cli`.

### File Locations

- Create the new ranking package under a top-level `ranking/` directory.
- Add tests under `tests/`.

### Constraints

- Keep this story limited to the offline ranking service, report export, CLI, and tests.
- Do not add database models, API endpoints, dashboard wiring, scheduling, or authentication.
- Do not modify the live runtime control plane or report persistence flow.
- Use stdlib for JSON/CSV serialization and timezone-aware UTC timestamps.

### Testing

- Run the ranking service and report export tests in the new package.
- Re-run the repository quality gates before closing the story.
