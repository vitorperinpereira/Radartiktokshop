## Story

**As a** project operator,
**I want** a CLI-first report builder that assembles and persists the weekly export payload from one scored pipeline run,
**so that** I can review a stable ranking summary with hooks and rationale without querying the database directly or adding dashboard write actions.

## Acceptance Criteria

1. `bin/radar.py` exposes a CLI-first report export command that resolves one completed scored run from an explicit `run_id` or the latest available scored week by default, then builds and persists a draft weekly report payload without introducing API or dashboard write flows.
2. The report builder assembles a deterministic payload from already persisted `pipeline_runs`, `product_scores`, `products`, `product_snapshots`, and existing explainability content only, including a week-level summary plus score-ordered ranked items with final score, classification, saturation/revenue context, and a first content hook with supporting rationale derived from stored data.
3. Report export persists the assembled report to `reports.report_payload` and stores one deterministic content-angle record per ranked product in the existing `content_angles` table, reusing current schema only and keeping the output scoped to the selected run/week.
4. Existing read history surfaces continue to work unchanged, but `/history/reports` now returns real generated payload content from persisted exports instead of fixture-only rows.
5. Tests cover pure report payload assembly plus sqlite-backed export persistence and history read behavior, including successful generation, latest-run defaulting, and empty/not-found failure handling where applicable.

## Dev Notes

### Implementation Context

- `docs/backlog.md` defines `CPR-020` under Epic 5 as "Implement report builder and export payload" with output limited to a weekly ranking summary plus hooks and rationale.
- `docs/project-tracker.md` marks `CPR-020` as the next pending work item after the completed API and dashboard read stories.
- `docs/implementation-plan.md` places report payload generation in Phase 4 after ranking/detail read surfaces and before release-smoke hardening.
- `docs/prd.md` requires a weekly ranking with explanation, classification, saturation risk, and a first content hook, while keeping the platform CLI-first.

### Existing Data and Surface Notes

- `services/shared/db/models.py` already defines `PipelineRun`, `ProductScore`, `ContentAngle`, and `Report`, so this story must reuse the current schema rather than introduce new tables or migrations.
- `services/reporting/read_service.py` already exposes ranking, product detail, and report history read behavior over persisted weekly output; `CPR-020` should add a builder/export layer alongside it.
- `services/scoring/aggregation.py` already persists explainability payloads, summary text, sub-scores, heuristics, and risk flags that should be reused directly for the report payload.
- `apps/api/main.py` and `apps/api/schemas.py` already expose `/history/reports`; keep those surfaces read-only and let them consume the new stored payloads without adding write endpoints.

### File Locations

- Keep `bin/radar.py` as the CLI control plane and extend it with the report export command.
- Add report builder logic under `services/reporting/`.
- Reuse `services/shared/config.py` for report timezone handling and `services/shared/db/session.py` for persistence wiring.
- Add focused regression coverage under `tests/unit` and `tests/integration` following existing scoring/API test patterns.

### Constraints

- Keep this story limited to `CPR-020`.
- Do not add dashboard write actions, API write endpoints, or publication workflow in this story.
- Do not add new schema columns, migrations, or new persistence tables.
- Do not invent report content that is not derivable from stored score output, explainability payloads, product metadata, snapshot context, or deterministic templates.
- Keep report generation scoped to a single completed run so weekly payloads do not mix outputs from different reruns.

### Testing

- Add unit coverage for deterministic content-angle generation and final payload structure.
- Add integration coverage for latest-run resolution, report persistence, content-angle persistence, and history reads of generated payloads.
- Verify repeated export updates the persisted draft for the same run instead of requiring manual cleanup.
- Run `npm run lint`, `npm run typecheck`, and `npm test` before closing the story.
