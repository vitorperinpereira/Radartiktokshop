## Story

**As a** project operator,
**I want** a deterministic final score aggregation step with persisted class bands and explainability payloads,
**so that** the weekly pipeline can emit reproducible final product scores after the three MVP agent runtimes complete.

## Acceptance Criteria

1. A deterministic aggregation runtime combines the three MVP agent scores (`Trend`, `Viral Potential`, and `Creator Accessibility`) with deterministic monetization and saturation heuristics using the formula defined in `docs/scoring-model.md`.
2. Final scores are mapped into class bands using the thresholds in `docs/scoring-model.md`, and the persisted `classification` is derived deterministically from those bands.
3. Each aggregated result produces an explainability payload aligned with `docs/scoring-model.md`, including raw sub-scores, applied weights, applied penalty values, merged risk flags, and human-readable explanation content (`summary`, `strengths`, `weaknesses`, `evidence`).
4. Aggregated results are persisted in `product_scores` using the existing schema fields already defined in `services/shared/db/models.py`, including the three agent score columns, `saturation_penalty`, `revenue_estimate`, `final_score`, `classification`, and `explainability_payload`, without introducing new schema columns in this story.
5. The weekly LangGraph flow executes final aggregation after Creator Accessibility, the weekly run summary/CLI path exposes final-scoring metrics when surfaced by the run, and tests cover aggregation runtime behavior plus weekly graph, weekly run, and smoke/CLI integration where applicable.

- `docs/scoring-model.md` defines the deterministic MVP formula, class bands, explainability requirements, and the expectation that scoring weights remain versioned per weekly run.

### Data and Persistence Notes

- `services/shared/db/models.py` already defines `ProductScore` with the persistence fields needed by this story: `trend_score`, `viral_potential_score`, `creator_accessibility_score`, `saturation_penalty`, `revenue_estimate`, `final_score`, `classification`, and `explainability_payload`.
- The same model also indexes `product_scores` on `(week_start, final_score)`, so this story should reuse the existing ranking-oriented persistence contract rather than introduce new schema shape.
- The schema does not currently expose a dedicated `monetization_score` column. If the deterministic monetization heuristic needs to be preserved separately from `revenue_estimate`, keep that detail inside `explainability_payload` while reusing `revenue_estimate` for the schema-level monetary proxy already available.

### File Locations

- `services/scoring/` is the intended home for deterministic aggregation, classification, and explainability policy code.
- `services/orchestration/state/weekly_state.py` currently tracks the three agent result collections and should be extended for final aggregation state and summary counters.
- `services/orchestration/graphs/weekly_graph.py` currently wires `extract_features -> run_trend_agent -> run_viral_potential_agent -> run_creator_accessibility_agent -> finalize_success`; insert final aggregation in that sequence.
- `services/orchestration/weekly_run.py` currently exposes only feature-extraction and per-agent summary metrics; extend it so final scoring metrics are available to the weekly-run path and CLI consumers.
- Existing regression entry points already live in `tests/unit/`, `tests/integration/test_weekly_graph.py`, `tests/integration/test_weekly_run_service.py`, and `tests/smoke/test_cli_weekly_run.py`.

### Constraints

- Keep this story limited to `CPR-017`.
- Do not add dedicated Saturation, Monetization/Revenue, or Content Angle agent runtimes in this story; `docs/scoring-model.md` explicitly marks those as deferred or deterministic MVP heuristics.
- Do not change the database schema in this story; persist through the fields already present in `product_scores`.
- Do not expand into API, report, or dashboard work from `CPR-018`, `CPR-019`, or `CPR-020`.
