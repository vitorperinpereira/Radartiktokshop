## Story

**As a** project operator,
**I want** the Streamlit Weekly Radar and Product Drilldown pages to read persisted ranking and product-detail data,
**so that** I can inspect the scored weekly output through the dashboard without relying on mock data or querying the database directly.

## Acceptance Criteria

1. `apps/dashboard/pages/1_weekly_radar.py` stops using cached mock data and renders the Weekly Radar from persisted ranking data exposed by the current read surface (`/rankings` or an equivalent thin dashboard read adapter), preserving a read-only, score-ordered table for the latest available scored week by default.
2. The Weekly Radar page preserves the current read-only operator workflow with live data: summary metrics, category filtering, hide-high-saturation filtering, and an obvious path into Product Drilldown using real product identifiers from the ranking response; it also handles empty-result and read-failure states without crashing the dashboard.
3. `apps/dashboard/pages/2_product_drilldown.py` stops using hard-coded product names and static reasoning, and instead renders one selected product from persisted detail data exposed by the current read surface (`/products/{product_id}` or an equivalent thin dashboard read adapter), including product metadata, final score/classification, agent sub-scores, saturation/revenue context, and any available latest snapshot metadata.
4. Product Drilldown renders stored explainability content from `score.explainability_payload` and existing response fields only; it must not generate or require report-builder/export content, publication workflow content, or new write actions in the dashboard.
5. Dashboard changes remain limited to Weekly Radar and Product Drilldown. The existing Pipeline History surface may remain as-is, and no release-smoke E2E coverage or report builder/export scope is added in this story.
6. Tests cover the dashboard read integration for Weekly Radar and Product Drilldown using the current persisted API/read contracts, including successful live-data rendering plus empty, error, and not-found handling where applicable.

## Dev Notes

### Implementation Context

- `docs/backlog.md` defines `CPR-019` under Epic 5 as "Build Streamlit ranking and product detail views" with output limited to a read-only dashboard with filters and details, and it depends on `CPR-018`.
- `docs/project-tracker.md` marks the current phase as "Dashboard and report surface prep", shows `CPR-018` as complete, and keeps `CPR-019`, `CPR-020`, and `CPR-022` pending. This story should stay narrow so report builder/export and release-smoke work remain available for their follow-up stories.
- `docs/implementation-plan.md` places Streamlit ranking and detail views in Phase 4 after the read endpoints and before the report payload builder.
- Story 1.11 already delivered the read-only API surface for ranking and product detail backed by persisted weekly scoring output. `CPR-019` should consume that surface rather than add a second database-specific read path in the dashboard.

### Existing Dashboard Surface

- `apps/dashboard/app.py` is the current read-only Streamlit entrypoint and already advertises Weekly Radar, Product Drilldown, and Pipeline History in the navigation copy. Keep that overall dashboard shape intact.
- `apps/dashboard/pages/1_weekly_radar.py` is currently a mock page backed by `load_mock_data()`, with category multiselect, `Hide High Saturation Risk`, three summary metrics, a dataframe, and a caption that pushes the user toward Product Drilldown.
- `apps/dashboard/pages/2_product_drilldown.py` is also mock-driven, with hard-coded product names, static metrics, a static "Content Angle Hook", and tabs for Trend, Viral Potential, and Accessibility reasoning.
- The project already has a mock pipeline-history surface, but the user explicitly scoped `CPR-019` to ranking and product detail views. Do not expand this story into pipeline-history rework.

### API / Read Contract Notes

- `apps/api/main.py` already exposes `/rankings` with optional `week_start`, `limit`, `category`, `classification`, and `hide_high_saturation`, plus `/products/{product_id}` with optional `week_start`. Reuse these read-only interfaces or a thin dashboard-side wrapper over them.
- `services/reporting/read_service.py` resolves the latest scored week when `week_start` is omitted for ranking requests and returns the latest persisted score row for a product when detail is requested without a week override. This gives the dashboard a stable default state without new write-side logic.
- `apps/api/schemas.py` defines the ranking fields already available to Weekly Radar: `product_id`, `run_id`, `week_start`, `title`, `brand`, `category`, `subcategory`, `image_url`, `final_score`, `classification`, `trend_score`, `viral_potential_score`, `creator_accessibility_score`, `saturation_penalty`, `revenue_estimate`, `saturation_risk`, `risk_flags`, and `summary`.
- Product detail already returns `product`, `score`, and `latest_snapshot`. `score.explainability_payload` is the source of truth for strengths/weaknesses/evidence-style UI, while `latest_snapshot` can surface captured price, rating, orders estimate, commission rate, source name, and source record id when available.
- There is no report-builder/export payload in the product detail response. If the current drilldown mock includes content that cannot be sourced from `product`, `score`, or `latest_snapshot`, replace it with persisted explainability data rather than extending the API surface here.

### File Locations

- Keep `apps/dashboard/app.py` as the dashboard entrypoint.
- Implement Weekly Radar changes in `apps/dashboard/pages/1_weekly_radar.py`.
- Implement Product Drilldown changes in `apps/dashboard/pages/2_product_drilldown.py`.
- If a reusable read-only helper is needed, add it under `apps/dashboard/` so the Streamlit pages stay thin and the API/read-service contract remains the single source of truth.
- Reuse the existing FastAPI read surface in `apps/api/main.py`, `apps/api/schemas.py`, and `services/reporting/read_service.py` rather than adding new persistence logic for `CPR-019`.

### Constraints

- Keep this story limited to `CPR-019`.
- Do not add report builder, export payload generation, publication workflow, or report-history UI work; that belongs to `CPR-020`.
- Do not add release smoke E2E or broaden test scope into weekly-run release validation; that belongs to `CPR-022`.
- Do not add new database tables, migrations, or write actions in Streamlit for this story.
- Do not rely on mock ranking or product data once the live read path is wired in.
- If the static Content Angle Hook cannot be backed by existing persisted/read-only data, remove or replace it with stored explainability content instead of inventing new copy.
